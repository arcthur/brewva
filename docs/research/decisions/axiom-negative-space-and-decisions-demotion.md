# Decision: Axiom Negative-Space Linkage And Decisions Demotion

## Metadata

- Decision: The axiom -> rule -> decision linkage is surfaced as a Tier-1 derived view generated from per-rule `(axiom N)` source tags in the project rule docs, never as a hand-authored negative-space map. An axiom with no enforcing rule is emitted as visible negative space, the `decisions/` archive stays immutable and provenance-only, and no new anti-patterns document is added.

- Status: accepted

## Decision Summary

- The rule docs carry the linkage at the source: each architectural bullet in `critical-rules.md` and `anti-patterns.md` ends in a `(axiom N)` source tag, so the axiom it serves is named where the rule lives rather than in a separate index.
- `docs/reference/axiom-enforcement.md` is generated by `script/generate-axiom-enforcement.ts` (sibling discipline to `generate-skill-navigation.ts`: build-time only, imports no `@brewva` package, fails closed on a dangling tag). One row per axiom: statement, enforcing rules, precedent decisions parsed from each accepted record's `## Axioms` section.
- An axiom whose enforcing-rule set is empty is emitted as explicit negative space, not suppressed, so an asserted-but-unenforced axiom stays visible instead of requiring a spelunk of the archive.
- The hand-authored negative-space map from the framing draft was rejected: it would be a second source for "which rule enforces which axiom," the drift surface axiom 18 forbids, and generation makes faithfulness a tautology. A new `design-anti-patterns.md` was also rejected.
- The 125 immutable decision records stay untouched and provenance-only; the view re-grounds them through a precedent-decisions column instead of a prose demotion ceremony, so demotion happens as a projection.
- `test/fitness/docs/axiom-enforcement-fresh.fitness.test.ts` guards the view with regenerate-and-diff plus referential-integrity assertions (every axiom has a row, every tag resolves), reusing the skill-navigation freshness mechanism rather than a new one.

## Axioms

These obey `docs/architecture/design-axioms.md`:

- Obeys `Descriptive metadata derives views, never authority` (axiom 18): the rule docs and axiom list are the source of record, and `axiom-enforcement.md` is a pure projection that only changes when a `(axiom N)` tag changes and is regenerated. The view never feeds runtime or authority back the other way; a hand-authored map would have created the reconciled-by-hand second authority this axiom exists to forbid.
- Obeys `Documentation hierarchy follows authority hierarchy` (axiom 14): the derived view sits below the rule docs and axioms it summarizes, and surfaces unenforced axioms as negative space without widening any rule's authority.
