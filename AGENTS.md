# Brewva Agent Guide

## Purpose

- This file is the short repository map for agents working in `brewva/`.
- Keep it current and action-oriented. Long-form project rules live in `skills/project/shared/*.md`; design detail lives in `docs/**`; code and tests remain authoritative.
- Priority order: hard invariants, working doctrine, workflow gates, verification, then lookup.

## Repo At A Glance

- `Brewva` is a Bun + TypeScript monorepo for an AI-native coding-agent runtime.
- Workspace packages live under `packages/*`; primary surfaces include `channels-telegram`, `cli`, `effect`, `gateway`, `ingress-telegram`, `mcp-adapter`, `provider-core`, `recall`, `runtime`, `search`, `session-index`, `std`, `substrate`, `token-estimation`, `tools`, and `vocabulary`.
- Release artifacts live under `distribution/brewva`, `distribution/brewva-*`, and `distribution/worker`.
- Support roots: `script/` for build and verification, `docs/` for design/reference material, `test/` for workspace coverage, and `docs/solutions/**` for repository-native precedent.
- Project guidance is authored in `skills/project/shared/*.md` with metadata-only `strength` and `scope` frontmatter. It is context/provenance metadata, not runtime authority.

## Hard Invariants

- Preserve the user-facing `brewva` command, help surface, launcher metadata, and distribution smoke checks.
- Keep the public runtime root four-port based: `identity`, `config`, `tape`, `kernel`, `model`, `start`, `turn`, and `close`.
- `createBrewvaRuntime(...)` requires `physics: { mode: ... }`; do not introduce default-mode fallbacks, implicit providers, or `EMPTY_PROVIDER` turn paths.
- Do not reintroduce `root`, `hosted`, `tool`, `operator`, `authority`, or `inspect` on `createBrewvaRuntime(...)`.
- Do not add private runtime construction, Tape commit, or Effect semantic service seams; new code must target the four-port runtime or a package-owned control-plane adapter.
- Runtime tests that still need historical hosted/operator fixtures must import them from `../../helpers/runtime.js`, not from a runtime package subpath.
- Keep workspace imports on package entrypoints; do not reintroduce local alias schemes or cross-package relative imports.
- Keep public root exports narrow. Repo-owned implementation seams stay under documented internal entrypoints.
- Keep managed tools capability-scoped and fail-closed when a runtime capability is undeclared.
- Keep runtime execution receipt-based, replay-first, and recoverable through existing WAL/event/proposal boundaries.
- Keep SQLite session index state rebuildable and non-authoritative; event tape remains replay authority.
- Keep search tokenization centralized in `@brewva/brewva-search`; do not add package-local tokenizers or silent Chinese-tokenizer fallbacks.
- Use Bun for build and test. Baseline: Bun `1.3.12`, Node `^20.19.0 || >=22.13.0`, ESM, strict TypeScript.
- Detailed invariant context is in `skills/project/shared/critical-rules.md`, `skills/project/shared/package-boundaries.md`, and `skills/project/shared/anti-patterns.md`.

## Working Doctrine

Operating principles distilled from repeated incidents in this repository. They apply equally to feature work and bug fixes.

- **Evidence over inference; durable layers over presentation layers.** Start every investigation from persisted evidence â€” tape events, WAL contents, execution audits, debug traces â€” never from UI output alone. Presentation layers and fallback paths systematically misclassify failures (stale config surfacing as connection errors, transient provider failures presenting as deadlocks, entitlement rejections masked by silent model fallback). Treat the apparent category of a symptom as unverified until durable evidence confirms it.
- **When layer-by-layer static review says "correct" everywhere, write a full-chain reproduction.** Multiple defects here passed static audit at every individual layer and surfaced immediately under one end-to-end integration test. For new features, stand up end-to-end verification first, then argue unit-level correctness. Re-reading code to convince yourself it should work is the signal to stop and write the reproduction.
- **One symptom rarely means one defect.** Several incidents were stacked, independent defects behind a single report; fixing the first plausible root cause and stopping is the most common way to leave a bug alive. Acceptance is the original symptom disappearing on the real path, not a reasonable-sounding cause being fixed.
- **Before touching any shared surface, enumerate every producer and consumer.** Render/interaction projection drift, boot-time setters silently overriding user configuration, and bindings shadowed by higher-priority layers are one disease: the cost of changing shared state, wire contracts, setters, or key bindings lives in the readers and writers you did not look at. Pull the full consumer list first; prefer a single shared projection over parallel views of the same state.
- **Distrust intermediaries; done means every gate empirically green.** Subagent findings are leads, not conclusions â€” verify against primary evidence before acting on them. Tool signals can be environment artifacts (shell wrappers mutating commands, default test timeouts, stale incremental-build state, stale worktree dependencies) â€” rule those out before trusting a failure. Completion requires `bun run check` and the full `bun test` suite both actually run and green; their coverage does not overlap.
- **Isolate workspace identity â€” every file operation and gate targets the one assigned worktree.** This checkout runs many worktrees and the repo root is itself `main`'s, so a root-absolute path or a `cd` to root silently writes edits and commits to `main` (an outward action here). The subtle failure is a false-green gate: edits made through root paths while gating in the feature worktree pass against the unedited base, because the worktree's `@brewva/*` symlinks mask the split, so verification tested nothing you wrote. Pick one worktree and one absolute-path prefix for both edits and `cd`; mechanics and recovery are in Workspace & Forensics.
- **Every writer to persisted state must separate user intent from machine re-application.** A setter shared between a user action and a boot- or install-time "read and re-apply" pass lets the restore path silently overwrite the user's choice: CLI theme persistence was dead under the default config because boot ran the persisting setter every startup, and a `postinstall` `deepMerge` of the default template resurrected removed config fields on every install (later surfacing as a bogus connection error). Boot and install paths seed-if-absent and restore through a persist-free path (`value = persisted ?? boot`); before shipping persistence, grep every caller of the setter and run the mental test "user changes it, restarts or reinstalls â€” is it still their value?".
- **Classify a failure as permanent or transient at its source and preserve that classification across every boundary.** Retry transient failures (connection-refused, rate limits) with bounded backoff; never retry a permanent one (a rejected entitlement, a stale-config refusal) and never let it surface as a transient hang or a deadlock. The recurring defect is a `retryable` signal dropped as it crosses a seam â€” rebuilt from a bare error string, unwrapped off its carrier, or re-derived without walking the `.cause` chain â€” which turns fail-fast into infinite retry or a masked symptom; an explicit `retryable === true` is authoritative over any message heuristic.
- **Scope every reference to externally-owned state to the lifecycle of its carrier.** A handle owned by a remote or connection authority â€” a server-side `previous_response_id`, a provider session, a cache handle â€” is valid only while its carrier lives, and reused past that it is silently treated as new rather than refused. A connection-scoped continuation id that outlived its idle WebSocket made the backend treat each resumed turn as a fresh session and evaporated the entire history with no error at all. Bind such handles to the connection or session id, invalidate on mismatch, and tear them down on replace, rewind, compaction, and reconnect.
- **Converge on a single authoritative writer and source of truth; derive every other view from it.** State with multiple writers or parallel sources is the bug class behind ordering drift, phantom rows, lost updates, and double commits â€” the fixes here repeatedly ended at one tape owner, one turn entry point, one ordered transcript, with the SQLite session index explicitly non-authoritative behind the replay tape. This is the write-side twin of preferring one shared projection: a single writer is a deliberate terminal state, not a temporary simplification, and new code adds a derivation rather than a second source.
- **Report only what a real run demonstrated; distinguish built from verified.** Keep tense honest in commits and summaries â€” "implemented", "scaffolded", or "submitted" is not "run", "verified", or "passed", and a scoreboard is not a played game (one arc narrated a scored evaluation loop that its empty corpus had never run). Per-change reviews structurally miss this cumulative over-claim; a final holistic pass over the whole arc against its original intent catches it, and the correction is tense discipline, not more code. Independent adversarial review stays non-optional â€” reading the same code, it repeatedly caught "green on synthetic, dead on real" blockers that self-review did not.
- **Name each durability and delivery guarantee honestly, and enforce the write discipline at every call site â€” not just the flagship logs.** Distinguish process-crash from power-loss durability and treat effect delivery as at-least-once with best-effort dedupe, never exactly-once (the vocabulary encodes exactly this: `DURABILITY_LEVELS` = `process_crash`/`power_loss`, `EFFECT_DELIVERY` = `at_least_once`), so every consumer of a replayed effect must be idempotent. Persist before mutating memory, fsync at commit and turn boundaries, and tolerate a single torn trailing line by truncate-on-load or by writing through tmp + rename + dir-fsync. The recurring failure is a durable-write helper that exists but is not applied where it matters â€” a control tape or credential vault that skips `fsync`, an integrity check hardcoded to "ok" that masks corruption â€” so route durable writes through the shared helper and let a fitness forbid raw `writeFileSync`/`appendFileSync` on durable paths.
- **Consolidate only what is genuinely identical or safely guardable; keep divergent look-alikes apart.** Migrate call sites to a shared primitive when they are byte-identical or provably equivalent, and defer superficially-similar ones whose semantics diverge â€” a clamp with different bounds, a truncate that counts its marker into the budget, an id scheme with a heterogeneous contract, an append-only reader with a concurrency or snapshot sub-problem. Over-merging couples independent change-vectors and can smuggle in a behavior change: a delegated retry-after clamp turned a `-999999` string that used to return `undefined` into `0`, and an append-only reader that truncates on load introduced a cross-process read/append race â€” a centralization must introduce no concurrency, durability, or semantic change, even a rare one. Back each shared primitive with a fitness that forbids the duplicated body structure, not just the name (a name ban is evaded by renamed copies), and enumerate every idiom of the pattern â€” `replace` vs `replaceAll` vs `split`/`join`, `.ts` and `.tsx`, both literal bounds of a unit clamp â€” or the guard has blind spots.

## Workflow Trigger Index

- Public runtime APIs, exported package surfaces, config schema/default semantics, persisted formats, WAL recovery, wire protocols, or user-facing CLI behavior: use `$implementation-strategy`.
- Multi-step, cross-package, refactor-heavy, or long-running work: use `$exec-plan`.
- Changes under `packages/**`, `test/**`, `script/**`, `package.json`, `tsconfig*.json`, `bunfig.toml`, or `.github/workflows/**`: run `$code-change-verification`.
- Changes under `docs/**`, `README.md`, or `test/docs/**`: run `$docs-verification`.
- Export, CLI, or distribution-surface changes: run `$dist-safety-gate`.
- Launcher or binary packaging behavior changes: run `$binary-packaging-verification`.
- Pi-specific tasks covering SDK, extensions, themes, skills, prompt templates, TUI, keybindings, providers, models, or packages: run `$pi-docs-sync`.
- Detailed workflow gate definitions and commands are in `skills/project/shared/workflow-gates.md`.

## Verification

- Default quality stack: `bun run check` and `bun test`.
- Property tests run inside `test:unit` and `test:contract` by filename; use `bun run test:property` for a focused property-only run and `bun run test:property:fuzz` for expanded runs.
- Docs stack: `bun run test:docs`; add `bun run format:docs:check` when Markdown formatting changed.
- Dist safety gate: `bun run test:dist`.
- Binary packaging verification: `bun run build:binaries` and `./distribution/brewva-linux-x64/bin/brewva --help | head -n 1`.

## Workspace & Forensics

Operating mechanics for this checkout. The reasoning behind them is in Working Doctrine; the specifics below are what actually bites. Items marked environment-observed are behaviors of this machine and its tooling, not repository invariants â€” verify before relying on them elsewhere.

- **Worktree isolation.** This checkout runs many git worktrees under `.claude/worktrees/`, and the repo root is itself a worktree pinned to `main`. Do every Write/Edit/Read and every Bash `cd` against the ONE assigned feature worktree's absolute path â€” a root-absolute path or `cd <root>` silently writes to `main`. Watch for the false-green: editing through root paths while gating in the feature worktree PASSES against the unedited base, because the worktree's `node_modules/@brewva/*` symlinks to root `src` so `@brewva/*` imports resolve to your edits while the worktree's own `test/` stays unedited â€” a green gate in a worktree whose `git status` is clean verified nothing. If `git status` shows no changes where you edited, you edited the wrong tree. A freshly created worktree needs `bun install --frozen-lockfile` before its imports resolve locally instead of to stale `main`.
- **Landing on `main` is an outward action (environment-observed).** In this setup an external mechanism (not a repo git hook â€” only `pre-commit` exists) rewrites `main` commits to sibling hashes and force-pushes `origin`, opaquely and with delay, even after a `--ff-only` merge. Treat a commit on `main` as a publish: never assume the hash is stable or the push private. Do feature work on a branch, ff-merge to `main` from the root via `git -C <root> merge --ff-only <branch>` (never `cd` into root to edit), keep a durable copy on the branch, and confirm remote state with a DELAYED `git fetch origin main` plus `git log --oneline origin/main..main`.
- **Forensics entry points.** Start every investigation from durable evidence (Working Doctrine). The event tape is `<cwd>/.brewva/tape/<sessionId>.jsonl` with two layers â€” top-level `type` (`tool.proposed` / `tool.committed` / `turn.*`) and `type:"custom"` `payload.kind` receipts; copy a tape into a scratchpad pseudo-workspace to dissect rather than reading a live worktree. A stuck turn is `runtime.suspended{cause:provider_retry}` + `turn.ended{status:failed}` (read the `status` field â€” `turn.ended`'s `cause` is always `terminal_commit`); exec failures are in `tool_execution_audit`; human-vs-machine turns show in `manifest.provider.model` plus approval latency. `brewva inspect --diagnostic` and `brewva --print --backend embedded` surface `[config:error]` (stale-config rejections otherwise masquerade as connection errors); `BREWVA_KEYMAP_DEBUG=<file>` traces keymaps; `bun run report:context-evidence <root> [--session id]` and `analyze:advisory-receipts` read economics and receipts offline.
- **Operator-environment traps (environment-observed; may not apply to your shell).** Bare `rg` is wrapper-aliased to inject a `--replace` that rewrites matches to `n` and mimics file corruption â€” use `command rg --no-config` or the Grep tool, and never write `-rn` (it parses as `-r n`, the same corruption). The Bash sandbox black-holes internal IPs, and `git commit` emits a benign `could not read IPC response` while still committing.
