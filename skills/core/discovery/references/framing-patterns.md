# Discovery Framing Patterns

Use this reference when `discovery` needs help turning a vague request into a
problem statement that `plan` can use directly.

## Core Questions

Prefer questions that expose real pressure:

- What pain exists today without this feature?
- What are users doing instead right now?
- What is the smallest wedge that would still be valuable?
- What should explicitly wait until later?
- What would prove this framing was wrong?

## Output Shape Heuristic

Good discovery outputs usually include:

- the real problem statement
- concrete pains or unmet needs
- the recommended first wedge
- explicit non-goals
- a short seed that hands off cleanly to plan

## Anti-Patterns

- feature echo: restating the feature request without discovering the real need
- vision inflation: jumping straight to a platform roadmap
- false certainty: hiding key unknowns instead of listing them as planning inputs
- neutral framing: presenting several directions without a recommendation
